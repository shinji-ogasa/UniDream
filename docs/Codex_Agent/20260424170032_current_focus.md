# 指示: 現状整理とBC prior最適化の進め方

作成者: Codex
宛先: GLM
作成時刻: 2026-04-24 17:00:32 JST
種別: 指示

## 背景

UniDream は不要な旧実験コード、checkpoint、旧script、旧config、後方互換entrypointを削除し、現在は `World Model -> Behavior Cloning -> Imagination Actor-Critic -> Validation selector -> Test backtest` の本線パイプラインに絞っている。

現在の有効な入口は以下。

```bash
uv run python -m unidream.cli.train --help
uv run unidream-train --help
```

現在の主な責務境界は以下。

- `unidream/cli`: CLI entrypoint と引数処理
- `unidream/data`: OHLCV取得、特徴量、returns、oracle label
- `unidream/world_model`: Transformer world model
- `unidream/actor_critic`: actor、critic、BC pretrain、imagination AC
- `unidream/experiments`: pipeline stage、fold orchestration、selector、test実行
- `unidream/eval`: backtest、WFO、PBO、regime評価

直近のBC prior検証では、rate match系の調整が過補正になり、fold 4 で `long/short` どちらかへ寄る collapse 傾向が見えたため、その方向は一旦採用しない。次は、feature-stress regime と dual residual controller 周辺を、短時間probeから慎重に検証する。

## 仮説

M2未達の主因は、AC以前のBC priorが不安定で、stress regime における行動分布が collapse しやすいことにある。

改善仮説は以下。

- BC prior の target / residual / stress shift のかけ方を整理すれば、固定long/固定short collapseを減らせる
- dual residual controller は維持しつつ、stress regime でのみ効く補正に限定すれば、通常regimeの性能悪化を抑えられる
- fold 4 のような崩れやすいfoldで改善兆候が出ない案は、全fold正式実行に進めてもM2達成に繋がりにくい

## 実装範囲

変更してよい範囲:

- `unidream/actor_critic/`
- `unidream/experiments/`
- `unidream/eval/`
- `configs/*dualresanchor*.yaml`
- 必要最小限の `README.md` / `docs/` 更新

原則として触らない範囲:

- `unidream/data/` のデータ取得仕様
- `unidream/world_model/` のモデル構造
- `uv.lock` を伴う依存追加
- 既存のtest split評価条件

責務分離の方針:

- CLI引数処理を学習ロジックに混ぜない
- config読み込み、BC prior計算、AC fine-tune、評価集計を同じ関数に押し込まない
- stress regime 判定と policy output 補正は、どちらが責務を持つか明確にする
- 一時実験コードは本線に混ぜず、configで切り替えられる狭い範囲に閉じ込める

## 削除方針

以下は復活させない。

- top-level `train.py` などの旧entrypoint
- `train_ppo.py`、`train_qdt.py`、risk-controller、event-controller系の旧実験
- source-rollout系の旧script / 旧config / 旧docs
- audit専用の旧probe module
- checkpoint、cache、logなどの生成物

実装中に不要になった後方互換、旧config分岐、旧オプション、到達不能コードを見つけた場合は削除してよい。ただし、削除した内容はGLM報告の「責務分離と削除」に必ず列挙すること。

## 実行コマンド

まず、変更前後で最低限以下を確認する。

```bash
uv run python -m unidream.cli.train --help
uv run unidream-train --help
uv run python -m compileall -q unidream
```

短時間probeは、まず fold 4 を優先する。

```bash
uv run python -m unidream.cli.train \
  --config configs/medium_l1_bc_continuous_exec_shortmass_regimebias_shift15_blend625_bandtarget_tradeonly_dualresanchor_stresstri_shiftonly_s007.yaml \
  --start 2020-01-01 \
  --end 2024-01-01 \
  --folds 4 \
  --stop-after bc \
  --device auto
```

probeでcollapseが悪化しない場合のみ、BC後段またはAC込みの短縮実行に進む。いきなり全fold正式実行に進めないこと。

## 成功条件

短時間probeの成功条件:

- fixed long / fixed short collapse が悪化しない
- fold 4 の行動分布が極端な片側100%に寄らない
- BC prior の変更理由が指標とログで説明できる
- 実装が責務境界を壊していない

Train Check の成功条件:

- `collapse_guard = pass` を維持する
- fold 4 で OOS 指標の悪化が限定的、または改善兆候がある
- alpha / Sharpe / maxdd / win_rate のうち単一指標だけでなく、複数指標で改善または悪化抑制が見える

Formal Run に進む条件:

- fold 4 probe と短縮Train Checkの両方で、collapse回避とOOS改善兆候がある
- 実装差分が小さく、設定で再現可能
- test splitを使ったチューニングになっていない

## 撤退条件

以下のいずれかに該当する場合は、その案を捨てる。

- fold 4 で再び片側100%に近いcollapseが出る
- `collapse_guard` が fail する
- alphaだけ改善して Sharpe、maxdd、win_rate が大きく悪化する
- 実装がBC、AC、eval、CLIの責務を混ぜて複雑化する
- 同じ仮説で3回連続して同じ失敗をする
- 累計5回試してもM2条件に近づく兆候がない

撤退時は、何を試し、どの指標が悪化し、なぜ次に進めないかを明記すること。
